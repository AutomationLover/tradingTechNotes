# Linux 内核中的锁与“能睡/不能睡”——原理与选择指南

> 面向内核/驱动开发的“选锁心法”：先判断“能不能睡”，再看“临界区长短”。

---

## 1. 基本概念：临界区与“能睡/不能睡”

### 1.1 临界区是什么？

**临界区** = 被锁保护起来的那段代码，在这段代码里访问**共享数据**（全局变量、链表、队列、状态机等），要求同一时刻只能有一个执行路径进入。

典型形式：

- `lock();`
-     操作共享链表 / 队列 / 计数器
- `unlock();`

上锁到解锁之间的这段，就是“临界区”。

### 1.2 “能睡 / 不能睡”是什么意思？

“能不能睡”本质是问：**当前上下文是否允许调用 `schedule()` 或任何可能阻塞的 API**。  

- 能睡 = 可以调用睡眠/阻塞接口（比如 `mutex_lock()`、`wait_event()`、`msleep()`、`kmalloc(GFP_KERNEL)` 等），调度器可以切走当前线程。
- 不能睡 = 禁止调用任何可能导致调度/阻塞的接口，否则会破坏内核假设，甚至直接 BUG/死锁。

后面所有“锁的选择”都是先基于这个判断。

---

## 2. 典型“不能睡”的上下文

以下场景一律视为“不能睡”，只能用**自旋锁 + `GFP_ATOMIC`** 等非阻塞方式：

### 2.1 中断上下文

- 硬中断 handler（top half）
- softirq / tasklet

特点：

- 不是正常“进程/线程”，没有可供挂起/恢复的调度状态；
- 调度器不能把它当普通任务挂起，所以**不能睡**。

### 2.2 持有自旋锁期间

只要你已经 `spin_lock()` 了，就必须假定“不能睡”：

- 自旋锁语义 = 持锁者正在执行、其他人自旋等待；
- 如果持锁者在临界区里调用了会睡眠的 API：
  - A CPU：拿到自旋锁 → 进入临界区 → 调用了 `schedule()` 睡眠；
  - B CPU：来抢锁，一直自旋等锁释放；
  - 但 A 已经睡了，不会再主动解锁，系统直接死锁。

所以规则是：**自旋锁内严禁睡眠**。

### 2.3 其他特殊“不可睡”回调

一些文档明确标注 `cannot sleep` 的回调，比如：

- 某些 notifier 回调；
- 部分 trace hook、RCU 回调；
- 设备驱动内部的某些 hot path。

在这些回调里，也要当作“不能睡”对待。

---

## 3. 典型“能睡”的上下文

以下场景通常可以视为“能睡”，可以使用 **mutex/rwsem + `GFP_KERNEL`** 等睡眠型原语：

### 3.1 普通进程上下文（系统调用路径）

- 用户态调用 `read/write/ioctl` 等进入内核；
- 没有持自旋锁、没有关中断、没有特殊 `cannot sleep` 限制。

此时：

- 内核可以安全调用 `schedule()` 挂起当前进程；
- 可以使用 `mutex_lock()`、`down()`、`wait_event()` 等睡眠同步原语。

### 3.2 内核线程（kthread）

- 例如 `kworker/*`、你自己用 `kthread_run()` 创建的线程；
- 只要不在中断上下文、没持自旋锁，就可以睡眠。

典型模式：

- 循环：
  - `wait_event()` 等待条件；
  - 被唤醒后拿 `mutex` 处理一批任务；

### 3.3 workqueue / 线程化中断（threaded IRQ）

- workqueue 的 worker 在线程上下文中执行，可以睡眠；
- `request_threaded_irq()` 的 `thread_fn` 在内核线程中执行，同样可以睡眠。

这两者都是**官方推荐的方式**，用于：

- 把 top half/softirq 中“不能睡”的长逻辑，搬到能睡的上下文里；
- 然后在能睡的上下文里用 mutex/rwsem 等更合适的同步方式。

## 4. 自旋锁的原理（忙等锁）

### 4.1 基础思想

**自旋锁（spinlock）** 是一种**忙等锁**：拿不到锁就持续在 CPU 上“自旋”检查锁状态，而不是 sleep。

典型结构：

- 一个原子变量表示锁状态（空闲/占用）；
- `spin_lock()` 内部做：
  - 使用原子指令（`xchg`/`cmpxchg` 等）尝试把锁从空闲改成占用；
  - 成功 → 拿到锁；
  - 失败 → 进入循环，不断重试（自旋）。

`spin_unlock()` 则是把锁状态从占用改回空闲，并用合适内存屏障保证写入顺序对其他 CPU 可见。

### 4.2 自旋锁的特征

- 拿不到锁时，不 sleep，不调度，只是在当前 CPU 上忙等；
- 不依赖调度器，不需要任务状态管理；
- 适合：
  - 不能睡的上下文（中断/softirq/持锁）；
  - 临界区非常短（几十–几百纳秒、几个指令）的场景。

缺点：

- 临界区一旦变长，其他 CPU 会一直“烧核”自旋，非常浪费资源；
- 多 CPU 热点锁会导致 cache line 在 CPU 间不断来回 ping-pong，严重影响性能。

---

## 5. mutex 的原理（睡眠互斥锁）

### 5.1 基础思想

**mutex** = 会睡眠的互斥锁：拿不到锁时，把当前任务挂到等待队列，然后 `schedule()` 掉，等锁释放后被唤醒。

简化流程：

1. `mutex_lock(&m)`：
   - 若 `m` 空闲：
     - 标记当前任务为 owner，直接返回；
   - 若 `m` 已被持有：
     - 把当前任务加入 `m` 的等待队列；
     - 设置任务状态为不可中断/可中断睡眠；
     - 调用 `schedule()`，CPU 去干别的事。

2. `mutex_unlock(&m)`：
   - 清除 owner；
   - 从等待队列取一个任务，唤醒它；
   - 被唤醒任务重新参与调度，再去拿锁、继续临界区。

### 5.2 特征

- 拿不到锁时**不占用 CPU**，适合临界区可能稍长、有一定竞争的情况；
- 有调度/上下文切换开销；
- 不适合“特别短但频繁抢锁”的场景。

---

## 6. rwsem 的原理（读写睡眠锁）

### 6.1 基础思想

**rwsem（读写信号量）** = 支持读写区分的睡眠锁：

- 多个 reader 可以同时持有读锁（down_read），互不阻塞；
- writer 持有写锁（down_write）时独占，禁止任何 reader 进来；
- 写锁请求需要等所有读锁释放；锁释放时的唤醒策略由实现管理。

### 6.2 适用场景

- “读多写少”的场景：
  - 如路由表、一些基本配置表、page cache 元数据等；
- 读路径对性能要求较高，允许多个 reader 并发；
- 写路径较少，可以接受少量额外开销和等待。

本质上：**rwsem 是“读写版 mutex”，都是睡眠类原语，只是读者可以并发。**

## 7. 临界区短 vs 临界区长：为何影响选锁？

### 7.1 “临界区短”时，自旋锁更划算

定义上的“短”不必真的用纳秒去算，直观判断：

- 从 `spin_lock()` 到 `spin_unlock()` 之间，只做几件极简单的事：
  - 改一个 flag；
  - 增减一个计数器；
  - `list_add()/list_del()` 操作；
  - 入队/出队一个 skb 或任务。

这种临界区耗时非常短（几十〜几百纳秒级），即使有竞争：

- 其他 CPU 自旋几百纳秒就等到了锁；
- 自旋这点时间比“挂起→上下文切换→再唤醒”要便宜得多；
- 所以这种场景用自旋锁经济合算。

类比：前面的人只“伸手拿个东西就走”，你站在门口等几秒完全可以接受，不值得去排队领号。

### 7.2 “临界区可能较长”时，自旋锁非常浪费

“长”指的是临界区内包含以下任一情况：

- 循环处理大量数据（遍历大链表、拷贝大 buffer 等）；
- 调用多层复杂函数，执行时间不确定；
- 可能调用阻塞/I/O 操作（磁盘、网络、设备）；
- 使用 `kmalloc(GFP_KERNEL)` 或其他可能触发 reclaim 的分配。

这种场景下：

- 持锁线程可能需要几微秒甚至毫秒才能释放锁；
- 其他 CPU 在这段时间一直自旋，100% 占用核，却没有任何进展；
- 锁竞争多的时候，会看到 load 很高但有效吞吐不高，典型“burn CPU”。

改用 mutex/rwsem：

- 拿不到锁的线程直接 sleep，CPU 去做别的任务；
- 锁释放时唤醒一个等待者，继续执行；
- 内核整体吞吐和延迟稳定性会更好。

类比：前面的人在厕所里洗澡、打电话，这时候你还站在门口死等就纯属浪费时间；排队坐着等叫号（mutex/rwsem）要划算得多。

## 8. 选锁决策：一步步判断

可以整理成一张简明决策表。

### 8.1 第一步：判断上下文——能睡吗？

1. **不能睡**（中断/softirq、持自旋锁、文档说 `cannot sleep`）：
   - 只能用：自旋锁（`spin_lock*` 家族）、RCU、原子操作、per‑CPU 结构；
   - 内存分配用 `GFP_ATOMIC`；
   - 临界区务必短小。

2. **能睡**（普通 sys_call、kthread、workqueue、threaded IRQ）：
   - 可以用：mutex、rwsem、completion、wait_queue 等；
   - 内存分配用 `GFP_KERNEL`；
   - 之后再看临界区长短和读写模式。

### 8.2 第二步：看临界区特征

在“能睡”的前提下，再判断：

- 临界区很短、竞争不激烈、也确实不睡：
  - 自旋锁 / 原子操作 / per‑CPU 变量优先（尤其是极短热路径）。

- 临界区可能较长、包含 I/O 或不确定耗时：
  - 用 mutex（读写不多的情况下）；
  - 用 rwsem（明显读多写少的情况下）；
  - 如临界区特别大，还需要考虑拆分锁、按数据结构拆分、或搬工作到更合适上下文。

---

## 9. 总结：一句话记忆

用一句话收个尾：

- **先问：这里能不能睡？**
  - 不能睡 → 自旋锁 / 原子 / per‑CPU，临界区必须短。
  - 能睡 → 优先 mutex/rwsem，让等待者 sleep，避免烧 CPU。

- **再看：临界区短还是长？**
  - 极短临界区 + 竞争不太严重 → 自旋锁 OK；
  - 可能较长 / 有 I/O / 有不确定耗时 → 别自旋，换成 mutex/rwsem 或重构。

把这套规则和你已经掌握的：

- 中断 vs 进程上下文；
- GFP_KERNEL vs GFP_ATOMIC；
- workqueue / threaded IRQ / softirq；

放在一起看，你在挑锁/设计同步时的整体框架就比较完整了。
