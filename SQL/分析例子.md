# 针对 IMC 岗位，Linux 内核需要掌握哪些知识点

下面是结合那份 IMC Quantitative Microstructure Analyst / Performance Team 的 JD，总结出来的 Linux 内核重点知识清单，按与你的场景（HFT、网络性能、eBPF）相关程度排序。

---

## 1. 网络栈 & 收发包路径（最高优先级）

围绕“从 NIC 到应用、再从应用到 NIC”的完整链路，重点理解：

### 1.1 驱动 + NAPI / 中断模型

- RX/TX ring buffer、DMA：
  - 网卡如何用环形队列存放收/发包；
  - DMA 把包搬进主内存。
- 硬中断 → `NET_RX_SOFTIRQ` → `net_rx_action` → **NAPI poll**：
  - 硬中断只“叫醒”CPU，然后关闭进一步 RX 中断；
  - `NET_RX_SOFTIRQ` 中的 `net_rx_action` 调用各个 NAPI 的 `poll()`；
  - NAPI poll 从 RX ring 拉包、构造 skb、交给协议栈。
- 调优相关参数：
  - `net.core.netdev_budget`、中断 coalescing（收包合并）；
  - 高 PPS 场景中这些参数如何改变 poll 的 batch、CPU 利用率和 jitter。

### 1.2 协议栈数据路径（尤其 RX）

- 收包主路径（IPv4 举例）：
  - `netif_receive_skb` / `__netif_receive_skb_core`
  - `ip_rcv` → `ip_local_deliver` / 转发路径
  - `tcp_v4_rcv` / `udp_rcv`
- `sk_buff` 结构与生命周期：
  - `head` / `data` / `tail` / `end`；
  - 引用计数、clone、free 对 cache 和延迟的影响。
- GRO / GSO / TSO / LRO：
  - 所在层级（L3/L4）；
  - 对 PPS/吞吐 vs 延迟的 trade-off。

### 1.3 多队列 & 多核利用

- 多队列 RX/TX：
  - 一个 NIC 上 Q0..Qn；
  - 每个 queue 有自己的 IRQ、NAPI 实例。
- RSS / RPS / RFS / XPS：
  - RSS：硬件按 5‑tuple hash 把流分到不同 RX 队列；
  - RPS：软中断层面把处理转到指定 CPU；
  - RFS：根据“哪个 CPU 的应用在处理这个 flow” steer 收包；
  - XPS：发送时按 CPU 选 TX 队列，减少 TX 队列锁竞争。
- 这些机制如何帮助你：
  - 把特定 feed / flow 锁定到特定 CPU（和线程）；
  - 提升 cache locality；
  - 做良好的 CPU 负载均衡而不破坏延迟。

### 1.4 socket 层

- 发送路径：
  - `sys_sendto` / `sys_write` → `sock_sendmsg` → `tcp_sendmsg` / `udp_sendmsg`；
- 接收路径：
  - `sys_recvfrom` / `sys_read` → `sock_recvmsg` → `tcp_recvmsg` / `udp_recvmsg`；
- socket 队列：
  - `sk_receive_queue` / send queue；
  - backlog 队列、半连接队列对行为的影响。

### 1.5 fast‑path / bypass 技术的“位置感”

- DPDK、XDP、AF_XDP、Onload/VMA 大致绕过哪些层：
  - 哪些流不再走 `netif_receive_skb` / `tcp_v4_rcv` / `tcp_sendmsg`；
- 对内核工具的影响：
  - tcpdump / iptables / tc / 大多数 eBPF hook 在 bypass 流量上是否可见；
- 在 HFT 环境中：
  - 哪些组件用 bypass（feed handler、特定网关）；
  - 哪些仍走内核栈（监控、管理面、非极限路径）。

## 2. 调度器、CPU 亲和性 & 中断（控制 jitter）

JD 里提到 “ultra‑low latency environments / OS architecture”，这部分是为了**解释 tail latency 的来源**。

### 2.1 调度基础（CFS / RT）

- CFS：
  - 基本负载计算和时间片分配理念；
  - 调度延迟与 CFS 调度周期的大致关系。
- RT 调度类（SCHED_FIFO / SCHED_RR）：
  - 行为（抢占普通任务、可能饿死其它线程）；
  - 什么时候用、什么时候危险。
- 抢占点：
  - syscall、软中断、内核临界区。

### 2.2 CPU 亲和性与隔离

- `sched_setaffinity` / `taskset`：
  - 如何把 feed handler / trading engine 固定在指定 CPU 上。
- `isolcpus`、`nohz_full`、`rcu_nocbs`：
  - 减少内核 housekeeping 对关键核的干扰；
  - 提升延迟可预测性。
- 将非关键任务（ksoftirqd、后台服务）迁移到非关键核。

### 2.3 中断与软中断行为

- 硬中断 vs 软中断：
  - 谁可以打断谁，谁不能睡眠；
- `ksoftirqd`：
  - 何时接管软中断（NET_RX_SOFTIRQ 等）；
  - 当 ksoftirqd 上网络负载高时，tail latency 会显著变差。
- 可观测性：
  - `/proc/interrupts`, `/proc/softirqs`；
  - eBPF / tracepoint（`irq:*`, `softirq:*`）。

## 3. 内存子系统 & NUMA（看延迟相关部分）

不要求内核内存专家，但要理解 **内存/NUMA 对延迟的影响**。

### 3.1 内存分配和 SLUB

- `kmalloc` / `vmalloc` 差别；
- 高频分配/释放导致：
  - page fault；
  - slab slow path；
  - cache 抖动；
- 高频路径里尽量复用对象 / 使用池。

### 3.2 NUMA

- NUMA node：
  - 本地 vs 远程内存访问延迟差异；
- NIC / CPU / 内存三者之间的拓扑：
  - NIC 在哪个 NUMA node；
  - feed handler / engine 线程跑在哪个 node；
  - 如何用 numactl / cgroups 让它们尽量同 node。

### 3.3 Huge pages & TLB

- hugepage 减少 TLB miss 的作用；
- 在 DPDK / 自研引擎 / 高频分配中，hugepage 几乎是标配；
- 通过 perf + eBPF 观察 TLB 和 page fault 行为对 tail latency 的影响。

## 4. 文件 & I/O 接口（相对次要，但要看得懂）

JD 提到“memory, network, and file I/O”，在 HFT 系统中：

- 撮合/行情核心路径避免阻塞文件 I/O；
- 但其他组件（日志、监控）如果绑错核，会影响关键线程。

需要理解：

- VFS 基本模型：
  - FD、page cache、writeback。
- `read` / `write` / `fsync`：
  - 在内核中可能如何阻塞；
- 用 eBPF / tracepoint 看：
  - 哪些线程在做大量同步 I/O；
  - 是否在关键时段发生了 writeback burst。

## 5. 内核可观测性接口：perf + eBPF hooks

JD 点名了 eBPF，说明希望你能用它们做 **系统级观测和诊断**。

### 5.1 基础工具

- perf：
  - CPU 利用率、hotspot callgraph；
  - cache miss、TLB miss；
- tracepoint / ftrace：
  - net:netif_receive_skb；
  - tcp:tcp_retransmit_skb；
  - sched:sched_switch；
  - irq:softirq_entry/exit 等；
- BPF maps：
  - HASH / ARRAY / PERCPU / RINGBUF；
  - 用于收集指标、事件。

### 5.2 关键 BPF 挂载点（网络/延迟相关）

- XDP / TC：
  - 在驱动 / qdisc 层对包做计数、过滤、时间戳；
- kprobe / kretprobe：
  - `napi_poll`、`netif_receive_skb`、`tcp_sendmsg`、`tcp_recvmsg`；
  - 用于打 key 函数的入口/出口时间戳，分解延迟路径；
- tracepoint：
  - `net:*`：网络事件；
  - `sched:*`：调度行为；
  - `irq:*` / `softirq:*`：中断/软中断行为。

### 5.3 写“低成本 BPF”的意识

- 使用 per-CPU map 减少锁竞争；
- 控制采样率，避免全量 trace；
- 避免在热路径使用 `bpf_trace_printk`；
- map 中只存回答问题所需的最小字段。

## 6. 与 kernel‑bypass 技术 / FPGA 的接口认识

JD 里提到：

- ultra‑low latency environments；
- kernel network bypass technologies。

你需要能：

- 清楚对比：
  - 标准内核网络栈 + NAPI；
  - DPDK / AF_XDP / Onload / FPGA；
- 知道 bypass 各自绕过了哪些内核路径：
  - 哪些 hook/工具（tcpdump/iptables/tc/eBPF）对它们还可见、哪些不可见；
- 明白架构组合：
  - feed handler / order gateway → bypass（DPDK/Onload/FPGA）；
  - 监控 / 非核心流程 → 仍用内核栈；
  - eBPF 主要观测剩余走内核的流量 & 系统行为。

## 7. 面试时的一句总括

可以把以上内容浓缩成类似表述（然后再展开细节）：

> On the kernel side, I focus on understanding the full RX/TX path — from NIC, interrupts, NAPI and `netif_receive_skb` through IP and TCP, all the way up to socket queues — and how scheduler, NUMA, and softirqs affect tail latency. I’m comfortable using perf and eBPF (XDP, TC, kprobes, tracepoints) to instrument key points like `napi_poll`, `netif_receive_skb`, `tcp_sendmsg` and `tcp_recvmsg`, and to reason about how kernel‑bypass technologies like DPDK or Onload change visibility for tools like tcpdump, iptables and tc.
