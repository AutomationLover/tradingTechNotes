# TCP 关闭连接时 RST/FIN 行为简要笔记

## 1. “缓冲区未读完就 close() 会发 RST”是谁规定的？

这条行为不是某个 RFC 条款逐字写出来的，而是 **TCP 实现层（如 Linux、BSD）的约定**，在不违反 RFC 语义的前提下做出的策略选择。

分层理解：

- 协议层（RFC 793、RFC 1122 等）  
  - 定义了 FIN、RST 的语义和 TCP 状态机（ESTABLISHED、FIN-WAIT-1、CLOSE-WAIT、TIME-WAIT 等）。  
  - 没有规定“应用调用 close() 时，内核在各种 corner case 下到底是发 FIN 还是发 RST”。

- Socket API 层（POSIX / BSD sockets）  
  - 定义了 `close()` / `shutdown()` 的语义：释放 fd、关闭读/写方向。  
  - 也没有强制规定“必须发 FIN 或必须发 RST”。

- 操作系统实现层（例如 Linux TCP 栈）  
  - 在 Linux 里，`close()` 最终进入 `tcp_close()`。  
  - `tcp_close()` 会根据当前 socket 状态、接收缓冲区是否有未读数据、`SO_LINGER` 配置等，决定：
    - 正常发送 FIN 并进入 TIME_WAIT，还是
    - 直接发送 active reset（RST），将连接置为 CLOSED。

所以，“接收缓冲区还有数据没读完就 close()，内核发 RST”是 **Linux 等实现中常见的策略**，用来表达“我这边对这个连接已经完全没兴趣了”，同时快速回收资源。这种行为与 RFC 对 RST 的语义是兼容的，但并不是某条 RFC 原文强行规定的。

---

## 2. 什么时候会出现“接收缓冲区还有数据没读完就 close()”？

这个现象本质是：**应用层提前放弃对对端后续数据的兴趣**，直接 `close()` 掉 socket。典型场景包括：

1. 应用超时或取消操作  
   - 发出请求后，没等把整个响应读完，业务逻辑就因为超时/用户取消而决定“不等了”：  
     - 上层逻辑直接 `close(sockfd)`；  
     - 此时接收缓冲区可能已有部分响应数据，甚至还在陆续到达。

2. 只关心写成功，不关心响应  
   - 例如某些“fire-and-forget”式客户端：  
     - 调用 `send()`/`write()` 发完数据；  
     - 根本不 `recv()`；  
     - 然后直接 `close()`。  
   - 若对端仍会发响应，这些响应就积在接收缓冲区里，从未被读取。

3. 错误处理或异常退出路径  
   - 某个错误分支中，异常处理逻辑直接 `close()`，没有先把已经到达的响应读掉。  
   - 或者线程/进程被粗暴终止，只做了简单的 `close()` 清理。

4. 误用 `close()` 代替半关闭  
   - 期望的语义是“我不再写，但还要继续读对端数据”，正规做法应是 `shutdown(SHUT_WR)`。  
   - 如果直接 `close()`，在 Linux 的常见实现里，接收缓冲区仍有数据且应用不再读取，会被视为“完全不要该连接了”，从而走 active RST 路径。

简单说：**应用不想再读数据，却直接 `close()` 把连接一刀切掉**，就可能触发“未读数据 + close() -> RST”。

---

## 3. 正常、合理的关闭过程应该怎样？

从协议语义和工程实践角度，比较“干净”的关闭方式是：在 **确保接收缓冲区的数据已经消费完** 的情况下再完全关闭。

### 3.1 只是不再写，还要读对端结果：使用半关闭（half-close）

典型流程（以客户端为例）：

1. 客户端发送完整请求：  
   - `send()` / `write()` 把请求发完。

2. 客户端声明自己不再发送数据（写方向关闭）：  
   - 调用  
     `shutdown(sockfd, SHUT_WR)`  
   - 内核发送一个带 `FIN` 标志位的 TCP 报文给对端。

3. 对端收到 FIN：  
   - 回复一个 `ACK`；  
   - 同时知道“对方不会再发数据了，但我仍可以发”。

4. 客户端继续读取对端响应：  
   - 循环 `recv()`，直到返回 0（表示对端也 FIN，读到 EOF）。  
   - 这期间接收缓冲区被应用主动消费完。

5. 客户端调用 `close()`：  
   - 此时既无未读数据，也无后续要发的数据，内核可以走标准 FIN/TIME_WAIT 流程，而非 RST。

服务端如果也需要对称地“写完然后读完”，可以采取类似流程：先处理请求并 `send()` 完响应，再 `shutdown(SHUT_WR)`、继续读，最后 `close()`。

### 3.2 从一侧完全放弃连接：RST 是可接受的语义

如果业务上 **明确不再关心对端任何后续数据**，主动 RST 在语义上并非错误：

- RST 告诉对端：“这个连接从我这边已经无效，再往这上面写就算错误”。  
- 对端写数据时会得到 `ECONNRESET` 等错误提示。

但这会带来两个后果：

- 对端应用通常把它当成“异常终止”，可能触发重试或报警；  
- 像 Nginx 这种服务端，会记录 `connection reset by peer` 之类的 error 级别日志。

如果你 **不希望业务侧看到这类“连接被对端 reset”异常**，就应该避免在“有未读数据”时直接 `close()`，而是遵循“读完 / half-close / 再 close”的流程。

## 4. 结合文章中的案例理解

在文章的案例 1 里：

- 三次握手、HTTP POST 以及 HTTP 200 响应都正常；  
- 客户端在 ACK 了 `HTTP 200` 后立刻发送 `RST+ACK`，跳过了正常的 FIN 挥手；  
- 服务端内核在 `recv()` 上收到 `ECONNRESET`，Nginx 记录 `connection reset by peer`。  

从实现视角看，这很可能是：

- 客户端在接收缓冲区仍有数据（比如 200 响应还没被用户态完全读走）的情况下执行了 `close()`；  
- 内核认为“应用已经不要这条连接了”，于是走 active reset 路径。

这不是 TCP 协议本身的“错误”，而是 **客户端应用层关闭策略的问题**。

---

## 5. 小结

- “缓冲区还有未读数据时 `close()` 导致内核发 RST”是主流 TCP 实现（如 Linux）的行为约定，不是单条 RFC“明文规定”的东西，但与 RFC 允许的 RST 语义一致。
- 这种情况通常出现在应用提前放弃读取、只关心写结果、超时/取消或误用 `close()` 代替 `shutdown(SHUT_WR)` 的场景。
- 更“干净”的关闭流程是：  
  - 写完数据 → `shutdown(SHUT_WR)` → 把接收缓冲数据读到 EOF → 最后再 `close()`。  
  - 这样可以避免在有未读数据时使用 RST 结束连接，减少像 `connection reset by peer` 这种误导性错误日志。
