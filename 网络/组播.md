# 针对组播需求应该准备哪些知识？（尤其是多播 UDP / ITCH-like）

结合之前岗位需求里提到的 **multicast**，以及你对 HFT / 低延迟的兴趣，可以从两大块来准备：

1. **Linux 上的组播网络实战**（socket/路由/IGMP）
2. **交易所行情场景下的多播 UDP 协议特性**（比如 ITCH/ITCH-like）

---

## 一、Linux 组播基础与实战

### 1. 组播基础概念

需要非常顺口地理解和讲清：

- 单播 / 广播 / 组播 的区别：
  - 单播：一对一。
  - 广播：一对所有（同一广播域）。
  - 组播：一对多（只发给订阅某组的成员）。
- IPv4 组播地址空间：
  - 224.0.0.0/4；
  - 区分本地控制（224.0.0.x）和 administratively scoped（239.x.x.x）地址段。
- 组播控制协议：
  - IGMP：主机向路由器声明加入/离开某个组；
  - PIM：路由器之间分发组播流（概念了解即可）。

交易所在机房里向某个多播组地址（例如 239.x.x.x:port）持续发送 UDP 行情流。订阅主机上的 feed handler 进程通过 IGMP 加入该多播组，交换机/路由器根据这些 IGMP 报文维护“谁订阅了哪个组”的表项，只在有订阅者的下游接口复制并转发这条多播流，因此一个行情流可以被多个主机高效共享，而不会在网络中产生不必要的重复单播流量。

---

### 2. Linux 多播 socket / IGMP 加入流程

要能书写和解释一个典型的多播 UDP 客户端流程（C 或 Python）：

- 建立 socket：
  - `socket(AF_INET, SOCK_DGRAM, 0)`
- 绑定本地端口：
  - `bind()`，通常是 `INADDR_ANY:port`
- 加入多播组：
  - 设置 `IP_ADD_MEMBERSHIP` / `IP_MULTICAST_IF`；
  - 使用 `struct ip_mreq` / `ip_mreqn`。
- 常用 socket 选项：
  - `SO_REUSEADDR` / `SO_REUSEPORT`：多进程监听同一端口；
  - `IP_MULTICAST_LOOP`：是否本机回环多播；
  - `SO_RCVBUF`：增大接收缓冲区防止丢包。

建议实际做一遍 demo：

- 启一个多播 sender（往 `239.x.x.x:port` 发 UDP）；
- 在另一进程/主机上 join 该组并打印消息；
- 用 `tcpdump` 看 IGMP 与组播数据包。

---

### 3. 多播在 Linux 内核路径中的位置 & 调优点

收包路径和单播网络一样走内核栈：

- NIC → NAPI → `netif_receive_skb` → `ip_rcv` → `udp_rcv` → socket。

多播特有关注点：

- 主机是否 join 了组（IGMP membership）；
- 中间是否有组播路由配置（如果跨 L3）。

性能/延迟调优方面：

- NAPI / RX 队列：
  - 高 PPS 多播（行情）情况下，`netdev_budget`、NAPI poll 行为；
  - RSS/RPS/RFS/XPS 队列与 CPU 亲和：避免所有多播流都打在同一个核心。
- socket buffer：
  - 系统层面的 `net.core.rmem_max` / `rmem_default`；
  - per-socket 的 `SO_RCVBUF`；
- 丢包排查：
  - `netstat -su` / `ss -u` 看 UDP 错误（`packet receive errors` 等）；
  - eBPF / perf 看 `udp_queue_rcv_skb` / `__skb_recv_udp` 等函数处的 drop 情况。

## 二、交易所行情相关的多播 UDP 特性（ITCH / ITCH-like）

对于 IMC/Jump/Citadel 这类岗位，“multicast” 几乎可以直接理解为“交易所行情多播”。

### 1. 多播行情的一般模式

典型特征：

- 多频道（channels）：
  - 按产品线/合约/价格等级拆成多条多播流；
- UDP 多播：
  - 单向：交易所 → 所有订阅者；
  - 高 PPS、小消息、顺序敏感；
- 你需要在本地：
  - 重组顺序（按 seq）；
  - 处理丢包、乱序；
  - 重建订单簿。

关键要点：

- 这不是轻量的控制流，而是可能达到数十万~百万 PPS 的持续流；
- 对 NAPI、队列、缓存、应用设计（无锁/低 GC）都有极高要求。

---

### 2. 序列号、丢包检测和重建

大多数 ITCH/ITCH-like 协议都包含：

- 每条消息一个 **sequence number**。

典型用法：

- 顺序检查：
  - 发现 seq 乱序或跳号；
- 丢包检测：
  - `seq_current > seq_last + 1` → gap；
- 重建策略：
  - 小 gap：发起 gap request / TCP replay；
  - 大 gap：重新 snapshot + replay；
  - 极端情况：完整重建订单簿。

从分析角度看：

- 记录 `(seq, recv_timestamp, channel, exchange)`；
- 用 SQL/pandas 分析：
  - inter-arrival 时间分布；
  - gap 的频率和时间分布；
  - gap 与 RTT/系统负载的相关性。

---

### 3. 延迟与抖动分析中，组播行情是 tick‑to‑trade 的起点

结合你前面学的 tick‑to‑trade：

- Tick 到达你 NIC 的时刻（多播行情包的 RX 时间戳）通常是：
  - tick‑to‑trade 计时的起点；
  - 你会在 NIC/hw timestamp、XDP、`netif_receive_skb`、`udp_recvmsg` 等点打时间戳；
- 多播 PPS 高时，对 NAPI/softirq/irq 亲和性配置非常敏感：
  - 某一段时间软中断被挤压，feed handler 就可能出现 tail-latency spike；
- 你会用 eBPF/tracepoint 观察：
  - `napi_poll` 耗时和 batch size 分布；
  - `NET_RX_SOFTIRQ` 行为；
  - CPU sched/irq 行为是否与 tick 延迟 spike 对齐。

## 三、建议的实践/准备路线

针对“组播 + ITCH-like + HFT”这条线，可以按下面实践路线准备：

1. **Linux 多播 socket demo**
   - 用 Python 或 C 写一个简单的多播 sender 和 receiver；
   - 观察 IGMP / 多播数据包，理解 socket 选项和加入组流程。

2. **行情流模拟器**
   - 写一个简单多播 sender：
     - 每秒 N 条消息，每条消息包含 seq 和发送时间戳；
   - 在 receiver：
     - 检测 gap；
     - 记录 `seq`, `recv_ts`；
     - 用 pandas 画 inter-arrival 和 P99 分布。

3. **结合 eBPF 观察 RX 行为**
   - 在 `napi_poll`, `net:netif_receive_skb` 等挂小 BPF：
     - 统计每 CPU 每秒处理的包数；
     - 记录 NAPI poll 耗时和 batch；
   - 看在高 PPS 模拟下系统表现如何。

4. **写一页笔记总结“多播行情在 HFT 系统中的角色”**
   - 为什么用 UDP 多播（单源多订阅、高带宽效率）；
   - 为什么需要 seq/gap fill 机制；
   - 多播带来的内核调优挑战（NAPI/IRQ/队列）。

## 四、面试用的一段英文总结示例

如果面试被问到“你对 multicast feeds 和 ITCH-like 协议的理解？”，可以大致这样说：

> In equity markets, exchanges typically disseminate market data over high‑rate UDP multicast feeds, often using ITCH or ITCH‑like protocols.  
> A feed handler joins the relevant multicast groups via IGMP, receives the UDP streams, decodes the binary messages, and reconstructs the limit order book.  
> This puts strong requirements on NAPI, RX queue and IRQ affinity tuning, as well as socket buffer sizing, to avoid packet loss and to keep tick‑to‑trade latency and jitter under control. Sequence numbers in the feed are used to detect loss and reorder, and to trigger gap‑fill or snapshot/replay when necessary.

然后结合你对 eBPF/XDP/NAPI 的掌握，解释你如何在内核路径上观察这些多播流，这样会和“分析 time‑series exchange network captures”这类 JD 非常吻合。
