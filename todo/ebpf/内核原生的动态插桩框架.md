# Linux 内核原生的动态插桩框架概览

Linux 提供了一整套“原生动态插桩框架”，允许你在**不修改内核源码、不重启机器**的情况下，对内核执行路径打钩子进行观测或轻量分析。常见的包括：

- kprobe / kretprobe
- tracepoint
- ftrace / function tracer（以及基于它的 fentry / fexit）
- perf events
- 以及配套的 uprobes 等用户态插桩机制

结合 eBPF，它们构成了一个强大的内核观测平台。

---

## 1. kprobe / kretprobe：任意内核函数上的“动态断点”

**kprobe** 和 **kretprobe** 是内核最灵活的动态插桩机制。

- kprobe：挂在某个内核函数或指令入口。
  - 每次执行到该位置，就触发你的 handler（或 eBPF 程序）。
- kretprobe：挂在函数返回点。
  - 可以看到返回值、函数执行时间（配合入口时间戳）。

**大致工作方式：**

1. 你注册一个 kprobe：“在函数 tcp_sendmsg 上插钩子”。  
2. 内核找到 tcp_sendmsg 的入口地址，在那插入陷阱指令。  
3. CPU 执行到此处时：
   - 触发陷阱 → 进入 kprobe 框架；
   - 调用你注册的 handler（或 BPF 程序）；
   - handler 执行完毕，恢复现场，继续执行原函数。

**在 eBPF 中的用法：**

- BPF 程序里，以段标记形式声明：

  - kprobe：  
    - SEC("kprobe/tcp_sendmsg")  
      - BPF_KPROBE(handle_tcp_sendmsg, struct sock *sk, struct msghdr *msg, size_t size)

  - kretprobe：  
    - SEC("kretprobe/tcp_sendmsg")  
      - BPF_KRETPROBE(handle_tcp_sendmsg_ret, int ret)

- 加载后，每次内核调用 tcp_sendmsg()，都会执行对应 BPF 程序。

**适合场景：**

- 网络路径观测：tcp_sendmsg、tcp_recvmsg、netif_receive_skb、napi_poll 等；
- 函数耗时分析：入口 kprobe 记 start_ts，返回 kretprobe 记 end_ts 做差；
- 临时验证假设：不用改内核源码，就能看参数/返回值/调用频率。

**缺点：**

- 依赖函数名和 ABI，内核升级时可能需要适配（CO-RE/BTF 很重要）；
- 原理上靠陷阱指令，比更现代的 fentry/fexit 多一点开销。

## 2. Tracepoint：内核预先埋好的稳定 hook 点

**tracepoint** 是内核开发者在关键路径显式埋下的事件点，既可用于 ftrace/perf，也可以挂 eBPF。

- 源码里通常是类似：

  - trace_netif_receive_skb(skb, ...);

- 这些点有“子系统:事件名”的名字，例如：
  - net:netif_receive_skb
  - sched:sched_switch
  - irq:irq_handler_entry
  - tcp:tcp_retransmit_skb
  - 等等。

**特点：**

- ABI 相对稳定：字段和意义由内核维护者保证；
- 比 kprobe 更安全：不直接依赖函数实现细节；
- eBPF 使用方式类似：

  - SEC("tracepoint/net/netif_receive_skb")

**适合场景：**

- 长期在线监控：调度行为、软中断、网络收发、块 I/O 等；
- 需要结构化数据：tracepoint 定义了 event 的字段类型，适合 CO-RE + BTF；
- 生产级观测：例如 tail 延迟抬高时看 netif_receive_skb 和 sched_switch 的行为变化。

---

## 3. ftrace / function tracer 以及 fentry / fexit

**ftrace** 是内核的函数级跟踪框架，最经典的是 function tracer：

- 可以在大量函数入口/出口插入跟踪逻辑；
- 和 tracefs 下的各种 tracer 一起使用（function, function_graph, sched_switch, irqsoff 等）。

在 eBPF 世界，内核进一步提供了 **fentry/fexit BPF 程序**：

- fentry：挂在函数入口；
- fexit：挂在函数返回点；
- 使用方式类似：

  - SEC("fentry/tcp_sendmsg")  
  - SEC("fexit/tcp_sendmsg")

**相对 kprobe 的优势：**

- 不再依赖 int3 之类陷阱，而是在函数 prologue/epilogue 更直接地钩住；
- 开销更低，延迟更稳定；
- 适合替代大量 kprobe 用法，尤其是在新内核上。

**适合场景：**

- 对很多函数做统一、低开销的入口/出口追踪；
- 长期在线使用，比 kprobe 更推荐（前提是内核和工具链支持）。

---

## 4. Perf events：采样 + 硬件计数器

**perf events** 提供了硬件/软件计数器和定期采样的机制：

- 硬件事件：
  - CPU cycles、instructions、cache-misses、branch-misses 等；
- 软件事件：
  - 上下文切换、page fault、sched events、tracepoint 等；
- eBPF 可以挂在 perf 采样事件上，在采样点执行自定义逻辑。

在性能/低延迟场景：

- 用于找出：
  - 热函数（hot functions）、热代码行（hot lines）；
  - cache/TLB 行为，评估 hugepage/NUMA 配置影响；
  - 结合 kprobe/tracepoint，将采样数据和结构化内核事件统一分析。

## 5. 其它相关插桩机制简要

### 5.1 uprobes / uretprobes（用户态函数插桩）

- 类似 kprobe，但作用于用户态 ELF 的函数：
  - 可以在你的 feed handler、撮合引擎、自研库等函数入口/出口上打钩子；
- 适合场景：
  - 应用级延迟剖析（比如业务函数耗时）；
  - 和内核级 kprobe/tracepoint 结合，形成端到端视图。

### 5.2 BPF LSM / cgroup hooks / socket hooks

严格来说更多是“策略 hook”而非调试插桩，但同属内核预留的扩展点：

- BPF LSM：在安全相关路径（open, exec, connect 等）上执行 BPF 逻辑；
- cgroup BPF：在 cgroup attach 的网络/系统调用 hook 上执行逻辑；
- socket BPF：在 socket 层过滤/负载均衡/观测。

在云原生 / 多租户 / 安全场景中，用于实现安全策略和细粒度网络控制。

## 6. 如何把这些和 eBPF / HFT 观测拼在一起？

从你关注的领域（Linux 网络性能 + HFT）来看，可以这样构建心智图：

- **数据来源（内核动态插桩）：**
  - kprobe/kretprobe：快速在任何内核函数上“插眼”；
  - tracepoint：用稳定事件做长期监控；
  - fentry/fexit：在新内核上以更低开销替代大部分 kprobe；
  - perf events：用采样 + 硬件计数器看 CPU/memory 行为。

- **逻辑承载（怎么处理这些事件）：**
  - 传统：内核 C handler + ftrace/perf 工具；
  - 现代：eBPF 程序，写 C（或 Go/Rust 封装），把观测数据写入 BPF map / ringbuf，由用户态 exporter 拉走。

- **典型用途：**
  - 网络栈观测：
    - 在 napi_poll、netif_receive_skb、tcp_sendmsg、tcp_recvmsg 上挂 kprobe/fentry；
    - 在 net:netif_receive_skb、tcp:tcp_retransmit_skb tracepoint 上挂 BPF。
  - 调度/中断观测：
    - 在 sched:sched_switch、irq:irq_handler_entry 等 tracepoint 上挂 BPF；
    - 用 perf 看软中断、上下文切换对 jitter 的影响。
  - 内核/应用端到端：
    - 内核侧 kprobe/tracepoint + 用户态侧 uprobes，拼出每条订单/行情的完整时间线。

**一句话总结：**

> Linux 原生动态插桩框架 = kprobe/kretprobe + tracepoint + ftrace/fentry/fexit + perf/uprobes，这些 hook 是内核官方提供的“插钩子”基础设施；而 eBPF 则给了你一种安全的、可编程的方式，把这些钩子变成高精度的观测和分析工具，特别适合你在 HFT 和高性能网络场景下剖析内核行为。
