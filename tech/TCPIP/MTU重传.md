# MTU 问题案例：NSX VXLAN 迁移后压测超时

## 背景：从传统 VLAN 网络迁移到 VMware NSX

公司原来后端服务跑在传统的三层网络 + VLAN 上，所有流量都在「裸 IP」网络里跑，物理网络 MTU 是 1500，应用压测一直很稳定。

为了后续做多租户隔离、微分段和自动化运维，网络团队规划了一次迁移：

**后端服务逐步上到 VMware NSX**，通过 VXLAN overlay 提供逻辑交换机和分布式防火墙能力。

### 迁移方案

- 客户端/压测机暂时还在传统网络（物理 VLAN）里；
- 后端应用服务器迁移到 NSX 逻辑交换机，通过 VXLAN 接入；
- 中间由 NSX Edge/DFW 负责东西向/南北向流量转发。

迁移后，业务功能上看都能访问，但在一次压测中暴露出问题。

---

## 现象：直连旧网络没问题，经 NSX VXLAN 压测就大量超时

压测团队在迁移前后的表现对比非常明显：

| 场景 | 表现 |
|------|------|
| **旧网络（物理 VLAN，未迁 NSX）** | 压测 HTTP QPS 很高，没有明显错误；抓包看到 RTT、服务器处理时间都在合理范围内。 |
| **迁移到 NSX VXLAN 后** | 压测脚本对每个请求配置了 1 秒超时；通过 NSX overlay 访问后端时，大量请求在 1 秒内没返回，报超时错误；但同一个后端服务，如果压测机直接走老的物理 VLAN（绕过 NSX overlay），又是好的。 |

简而言之：「直接打老网没问题，走 NSX VXLAN 就不行」，表面上看像是 NSX 性能不行，或者 overlay 有 bug。

---

## 抓包分析：DupAck + 重传，怀疑中途丢了特定大小的数据段

为了区分「应用慢」还是「网络丢」，我们在压测客户端一侧抓包（也可以在 NSX Edge 上抓），重点观察一条失败的 TCP 流：

**1. 三次握手正常** — SYN / SYN+ACK / ACK 都收发良好。

**2. 客户端发 HTTP 请求（POST），服务端 ACK 正常。**

**3. 服务端返回 HTTP 响应时，拆成了 3 个 TCP 段：**
- 段 1：长度约 1400 左右（示例）
- 段 2：长度略小
- 段 3：剩余部分

**4. 客户端只收到了后两个段，没有收到第一个大段，于是：**
- 反复发出重复确认（DupAck），确认号一直停留在「握手结束后」的位置；
- 抓包里看得到多次 DupAck 指向同一个期望的 seq。

**5. 服务端按 TCP 超时重传策略，对那个「首段」进行重传两次：**
- 时间间隔大约 200ms、400ms 这样指数退避；
- 但这两次重传，客户端一样收不到。

**6. 压测脚本 1 秒超时到达后**，客户端主动发 FIN/RST，终止连接并记录为 HTTP 超时报错。

### 分析结论

从这一条 TCP 流可以看出：

- 应用线程并不慢，服务端其实已经吐出 HTTP 200 了；
- 问题是：某一特定大小的 TCP 段，无论首次发送还是重传，都在路径中「消失」了；
- 客户端只收到了后面的小段，因此狂发 DupAck，服务端也尽职重传，但仍然白费。

这种现象非常符合**路径 MTU 问题**：只要报文尺寸超过某处链路的有效 MTU，就会被悄悄丢掉。

---

## 对比直连 vs 经 NSX：发现「最大 TCP 段大小」不同

对比两份抓包：直连旧网络（未过 NSX）的成功流量 vs 经 NSX VXLAN overlay 的失败流量。

| 路径 | 最大 TCP 段 | 表现 |
|------|-------------|------|
| **直连旧网络** | 约 1348 字节（示例） | 压测稳定，无 DupAck/重传 |
| **经 NSX VXLAN** | 约 1388 字节，比直连时大出 40 字节 | 1388 这一段无论原始还是重传都收不到；其他较小的段没问题 |

结合 NSX 的网络结构：**NSX VXLAN 本身就会在 IP 包外再套一层 VXLAN 头**（UDP + VXLAN header 等），相当于给每个包多穿一件「外套」，减少了有效可用 MTU。

| 环境 | 物理链路 MTU | 有效内层 MTU |
|------|--------------|--------------|
| **旧网络** | 1500 | 没有 overlay，IP/TCP 封完后刚好 1500 以内 |
| **NSX overlay** | 1500 | 加上 VLAN、IP、UDP、VXLAN 等封装头以后，承载内层 TCP 的有效 MTU < 1500 |

内层 TCP 还按照「老习惯」的 MSS 切段，比如切出 1388 这种「边缘尺寸」，一旦加上外层封装，就超过路径中某段链路的最小 MTU，被中途设备直接丢弃。

### 推断

通过 NSX VXLAN 的路径上，**内层 TCP 报文加上 VXLAN 封装后超过了中间链路的 MTU**，所以某个段（及其重传）一直被丢，引发 DupAck + 重传 + 应用超时。

---

## 根因：迁移到 NSX VXLAN 后，有效 MTU 未重算，应用仍按旧 MSS 发大包

1. **原物理网络 MTU 1500**，应用/内核根据这个环境形成了一套 MSS/分段行为，比如最大 TCP 段 1388 没问题。

2. **迁移到 NSX 后**，数据路径变成了「内层 TCP/IP + 外层 IP/UDP/VXLAN」：
   - 物理口还是 MTU 1500，但这 1500 里要包含外层封装头；
   - 导致给内层 TCP 的「有效 MTU」变小。

3. **但应用端/服务器端的 MSS 仍然是按「老环境」算出来的**，并没有针对 VXLAN overlay 重新调整。

4. **某些接近上限的内层 TCP 段**（例如 1388 字节）在加上外层封装后超出中间设备的 MTU，被丢弃。

5. 这正是抓包里看到的：同一尺寸的 Segment，无论首次发送还是重传，都消失在 NSX 路径里，而直连传统网络时则完全正常。

> **一句话：** 迁移到 NSX VXLAN 后，没有同步调整/校验有效 MTU 和 MSS，导致 overlay 路径上「理论 1500」实则装不下原来的大包。

---

## 解决思路（NSX 场景）

**1. 在物理网络上调大 MTU（比如 1600/9000）**
- 物理交换机 / 路由器 / 上联链路统一改成更大的 MTU；
- 确保「内层 TCP + 外层 VXLAN 封装」仍然落在物理 MTU 内。

**2. 在 NSX Edge / DFW / 中间网关上用策略调小 MSS**
- 类似 `iptables --set-mss`，在 NSX 边缘设备对握手中的 MSS 做 rewrite；
- 「暗箱操作」地把内层 TCP MSS 调小，例如 1360 或 1400；
- 强制两端以后发送更小的 TCP 段，以适配 VXLAN 后的有效 MTU。

**3. 在后端服务器或压测机上显式调小 MTU 或 MSS**
- 在虚机/容器上把接口 MTU 设置为 1400 左右；
- 或通过 sysctl / iptables / NSX DFW 策略限制 MSS。

### 案例要点

- 直连「旧网络」没问题，说明应用本身没慢，服务器也没问题；
- 一旦走 NSX VXLAN overlay，报文加上封装头就把有效 MTU 吃掉了一部分；
- 如果迁移规划里只考虑到「IP 能 ping 通」、「端口能 telnet 通」，但没做大包压测 / MTU 验证，就很容易在高压/大载荷场景下踩雷。

---

## Case 总结（适合 slide / 简历 bullet）

在一次从传统 VLAN 网络迁移到 VMware NSX VXLAN overlay 的项目中，我们发现：

- 旧网络下压测正常，但通过 NSX 转发时大量 HTTP 请求在 1 秒内超时。
- 抓包分析显示，握手和应用处理均正常，但特定大小的 TCP 段在 overlay 路径上始终丢失，伴随 DupAck 和重传。
- **根因：** NSX VXLAN 封装后降低了有效 MTU，而两端仍按传统网络的 MSS 发送大包，导致这些大包在中途链路被丢弃。
- **解决：** 通过在 NSX 边缘及物理网络上统一规划 MTU/MSS 并验证 Path MTU，最终解决了问题。
