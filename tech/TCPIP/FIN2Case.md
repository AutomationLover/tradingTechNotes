# TCP 连接关闭案例分析：RST vs FIN

**总览：** 案例 1 的问题在客户端用 RST 非正常结束连接；案例 2 是 Wireshark 展示误导，实际上是带 FIN 的正常挥手。

---

## 案例 1：Nginx 日志里的 connection reset by peer

### 关键结论

- 抓包显示：三次握手正常、HTTP POST 请求正常、服务端也正常返回了 HTTP 200。
- 在服务端发完 200，客户端 ACK 之后，客户端立刻发了 **RST+ACK**，直接把连接「掀桌子」断掉，而不是按 TCP 四次挥手用 FIN 正常关闭。
- 这个 RST 让服务端内核在 `recv()` 上收到 `ECONNRESET`，于是 Nginx 记录 `connection reset by peer`。

### 根因

- **问题点在客户端**：客户端代码/行为不规范，用 RST 来结束已建立的连接（典型场景是：接收缓冲区还有数据没读完就 `close()`，内核就发 RST）。
- **业务上**：这次 POST 已经被服务端成功处理，日志里虽然是 error 级别，但不是「事务失败」，而是「连接结束方式异常」。

### 一句话总结

> 不是 Nginx 或网络有问题，而是客户端用 RST 异常收尾，触发了 Nginx 的报错日志。

## 案例 2：看起来「只有一个 FIN」的挥手

### 表面现象

- 抓包里只看到后面两个报文：`FIN+ACK` 和最后的 `ACK`，好像整个连接关闭只有「一个 FIN」，不符合「四次挥手」的常识。

### 深入分析

- Wireshark 主界面的 Info 列展示的是应用层信息（比如 `POST /xxx`），不会把 TCP 标志位都展开。
- 选中那个 POST 报文，在中间的 TCP 详情里看，发现它本身就带了 `FIN` 标志位，也就是把数据报和 FIN 合并在一个包里（**piggybacking**）。
- 于是实际过程是：
  - 第一个报文：`POST ...` + `FIN`（发起端说「这是最后一批数据了」）；
  - 第二个报文：对端 `FIN+ACK`；
  - 第三个报文：最后的 `ACK`。

逻辑上依然是标准的「双方各发一个 FIN、互相 ACK」的挥手，只是第一个 FIN 被「藏」在应用数据包里。

### 结论

- 协议层没有问题，还是标准的 TCP 挥手；
- 问题在于观察和理解：如果只看 Wireshark 的 Info 列，很容易误以为少了一个 FIN，实际是 FIN piggyback 在数据包上，被展示风格「误导」了。

### 一句话总结

> 案例 2 没有真正的协议错误，是 Wireshark 展示方式让人误读，实际挥手完全符合规范，只是用了 piggyback 的 FIN。
