# Exchange-Specific Protocols：ITCH, OUCH, FIX 概要

在证券/高频交易场景下，常见三种核心协议分别负责不同功能：

- **ITCH：高速行情分发（读）**
- **OUCH：低延迟下单（写）**
- **FIX：通用金融消息标准（全能但相对慢）**

---

## 1. ITCH：行情分发协议（Market Data）

**定位**：由 Nasdaq 等交易所使用的二进制行情广播协议，用于分发极低延迟的订单簿更新。

**主要用途**

- 实时广播订单、成交、撤单等事件
- 客户端根据增量消息重建完整订单簿（Order Book）

**技术特性**

- 单向：交易所 → 客户端（只读，无下单）
- 消息格式：固定长度二进制，解析速度极快
- 延迟：纳秒 / 微秒级，常用于高频交易 (HFT)
- 典型传输封装：
  - UDP 组播 + MoldUDP64：实时行情主链路
  - TCP + SoupBinTCP：用于快照 (Snapshot/Glimpse) 和丢包重传 (Recovery)
- 工作模式示例：
  1. 启动时通过 TCP（Glimpse）拉取当前订单簿快照
  2. 切换到 UDP 组播持续接收增量更新
  3. 若通过序列号发现丢包，用 TCP 请求补发缺失消息

---

## 2. OUCH：低延迟订单入口协议（Order Entry）

**定位**：交易所原生的低延迟下单协议，通常与 ITCH 配合使用。

**主要用途**

- 发送订单指令：新单、撤单、改单
- 接收执行报告：成交、拒单、状态更新

**技术特性**

- 双向：客户端 ↔ 交易所
- 消息格式：固定长度二进制，字段少、语义简单
- 延迟：极低（原生接入，比 FIX 更快）
- 功能范围：通常覆盖基础限价单、撤单、改单等核心操作（不追求复杂业务语义）
- 典型使用场景：
  - 高频交易 (HFT)
  - 做市、套利等对延迟极度敏感的策略

---

## 3. FIX：金融信息交换协议（Financial Information eXchange）

**定位**：全球证券行业广泛采用的开放标准协议，覆盖交易生命周期的多种业务场景。

**主要用途**

- 下单与执行：订单提交、修改、撤单、执行回报
- 行情（部分场景）
- 交易后处理：分配、对账、清算、合规与监管报告

**技术特性**

- 通用性：几乎所有交易所、券商、OMS/EMS 系统都支持
- 消息格式：
  - 文本 Tag=Value 形式，如：
    - `35=D`：新订单（MessageType = New Order – Single）
    - 字段通过 SOH 分隔，易读易调试
- 延迟与性能：
  - 文本编码、字段较多，报文体积大
  - 序列化/反序列化开销大于 ITCH/OUCH，因此延迟相对更高
- 灵活性：
  - 支持大量业务字段、复杂订单类型（冰山单、TWAP/VWAP 算法指令、篮子交易等）
  - 易扩展，支持自定义 Tag，适合跨机构、跨市场对接

---

## 4. 对比总结：ITCH vs OUCH vs FIX

### 功能定位

- ITCH：只读行情通道，专注于订单簿更新
- OUCH：原生低延迟下单通道
- FIX：通用、多场景的业务协议，覆盖从下单到交易后处理

### 消息与性能

- ITCH：固定长度二进制，单向广播，延迟极低
- OUCH：固定长度二进制，双向请求-响应，延迟极低
- FIX：文本 Tag=Value，可读性强但体积大，延迟较高

### 典型组合方式

- **极致延迟 / HFT 场景**：
  - 行情：ITCH (UDP 组播 + 快照/恢复 TCP)
  - 下单：OUCH
- **跨市场 / 跨券商 / 复杂业务场景**：
  - 统一使用 FIX，简化系统集成与对接

---

## 5. 什么时候用哪一种？

- 如果你的目标是：  
  - 做 HFT、做市、套利，对每微秒都很敏感 → **ITCH + OUCH**
- 如果你的目标是：  
  - 对接多家券商 / 交易所，需要统一接口和复杂业务语义 → **FIX**
- 如果你只关心行情重建订单簿，不涉及下单 → **只用 ITCH 或其等价行情协议**

## UDP 组播在交易系统中的三个特别之处

在 ITCH 等行情协议中，UDP 实现几乎完全等同于 **IP 组播 (Multicast)**，其特殊性主要体现在以下三点：

### 1. 硬件级“零延迟”分发（Hardware Fan-out）

- 普通单播 UDP：有 N 个客户端时，服务器需要发送 N 份副本，同一条行情到达不同客户端会有先后差异。
- IP 组播：服务器只发送一份数据到组播地址（如 `233.x.x.x`），交换机在硬件层复制并转发给所有订阅端口。
- 结果：同一机房内所有订阅者几乎在同一微秒级别收到数据，有利于**延迟公平性**。

### 2. 必须配合“序列号”管理（Sequence Numbers）

- UDP 本身不保证可靠性（无重传、无乱序纠正），但交易订阅侧又不能容忍行情缺失。
- 解决办法：在 ITCH 报文载荷中加入单调递增的 **Sequence Number**。
  - 行情接收程序（Feed Handler）检测到跳号（例如 100 → 102）即可判断丢包。
  - 此时通过 **TCP（SoupBinTCP）** 向交易所请求补发缺失的报文（如 Seq=101），弥补 UDP 的不可靠性。

### 3. A/B 双路并行（A/B Feeds & Arbitration）

- 为了提高可用性与进一步降低依赖 TCP 回放的概率，交易所通常会提供两路完全相同的组播流：
  - Feed A 和 Feed B 走不同光纤与交换机路径。
- 客户端同时订阅 A/B 两路：
  - 正常情况下只消费先到达的一路，另一条作为冗余。
  - 一旦某一路出现丢包或抖动，可从另一条流补齐同一 Sequence，从而避免触发较慢的 TCP Recovery。
- 解析端需要做 **仲裁/去重**：基于 Sequence Number 保证只处理一次同一条逻辑消息。

## 组播在网络配置与软件解析上的高标准

在金融交易场景中，组播不仅是“能跑起来”，而是追求在**极端负载+极端低延迟**下仍然可预测、可靠。因此在网络与软件两侧都提出了很高要求。

### 一、网络侧：低延迟组播路由（Cisco/Arista）

1. PIM Sparse-Mode + Static RP  
   - 部署 **PIM-SM** 实现组播路由，并尽量使用**静态 RP**（而非 Auto-RP/BSR），降低控制平面不确定性。
   - 保证从 RP 到接收者路径固定、可预测，避免拓扑变化导致的时延抖动。

2. IGMP Snooping 与 Querier  
   - 开启 **IGMP Snooping** 确保只将组播流量转发到真正订阅的端口。
   - 配置稳定的 **IGMP Querier**，防止组播状态因超时被清除，导致行情中断。

3. 高可用与延迟可观测性  
   - 使用 **Anycast RP**、双上行、冗余核心等方式保障 RP/树根不成为单点。
   - 关注交换机缓存与队列：行情突发可能在极短时间打满队列，造成毫秒级但难以察觉的**微突发丢包**，需要通过队列监控与合理队列配置缓解。

4. 低延迟转发模式  
   - 使用支持 **Cut-through** 的交换机，减少每跳延迟。
   - 在同一 L2/L3 域内尽量减少中间转发节点，保证路径简单、延迟稳定。

---

### 二、软件侧：C++ 行情解析的性能与健壮性

1. 内核旁路与网卡优化  
   - 使用 **OpenOnload、DPDK、FPGA NIC** 等技术，将数据从网卡尽可能直接送达用户态内存，减少内核栈的上下文切换和拷贝。
   - 减少/屏蔽中断抖动（IRQ affinity、RPS/RFS 配置），保证接收线程的调度确定性。

2. 零拷贝与结构化解析  
   - 利用 ITCH 报文的固定格式，将接收缓冲区直接 `reinterpret_cast` 为结构体（配合 `#pragma pack(push, 1)`），避免逐字段解析。
   - 使用 CPU 原语或编译器内建函数做高效字节序转换（如 `__builtin_bswap*`），减少分支和函数调用开销。

3. A/B 仲裁与序列一致性  
   - 软件层需要实现对两路行情流的 Sequence 仲裁与去重逻辑：
     - 只处理第一条到达的某个 Sequence，丢弃同一 Sequence 的重复包。
     - 一旦检测到跳号，在触发 TCP Recovery 前先尝试从另一条流补齐。

4. 线程模型与 CPU 亲和性  
   - 将接收线程绑定到固定物理核（关闭超线程或避免和系统中断共用同核），减少调度抖动。
   - 使用无锁队列等数据结构在“接收”和“业务处理”之间解耦，保证接收端永远优先不阻塞。

---

### 总结

在高频金融场景中，**UDP 组播不仅仅是一个传输选项，而是一整套从交换机到 C++ 程序的系统工程**。  
它依赖组播路由（PIM/IGMP）的精细配置、交换机缓存和转发模式的严格控制，以及用户态行情解析器中对 Sequence、A/B 仲裁、内核旁路与 CPU 亲和性的极致追求，才能实现既**极低延迟**又**高可靠**的行情分发。
