# Linux 内核追踪框架概览：perf、ftrace、eBPF、SystemTap、sysdig

> 目标：搞清楚 Linux 内核/性能追踪的几大常用框架的 **定位、原理、优缺点和典型使用场景**，建立整体心智模型。

## 1. 为什么需要内核追踪框架

- 复杂系统中，性能问题/故障往往发生在 **内核与应用交界处**，仅靠 `top/ps/iostat` 等工具很难定位。
- 直接看内核源码、加打印：
  - 侵入性强、需要重新编译内核或模块；
  - 不适合线上环境。
- 内核追踪框架的目标：
  - 尽量 **低开销、动态、可控** 地观测内核事件；
  - 支持 **按需追踪** 某些函数、syscall、调度、内存、IO 等子系统；
  - 尽可能不影响生产业务。

常见技术路线：
- 静态探针（static tracepoints）
- 动态探针（kprobe/kretprobe, uprobes）
- 采样（sampling） vs 全量事件（tracing）
- 内核空间 vs 用户空间协同

## 2. ftrace：内核自带的函数级追踪框架

### 2.1 定位与特点

- **内核原生** trace 组件，从 2.6 时代发展而来。
- 核心设计：对内核函数调用、调度、IRQ 等进行追踪。
- 主要优点：
  - 内置于内核，无需额外安装内核模块或外部框架；
  - 提供多种 tracer：function、function_graph、sched、irqsoff 等；
  - 开销可控，可按 need 启用/禁用。
- 主要缺点：
  - 配置接口主要是 `tracefs`（`/sys/kernel/debug/tracing`），原始操作略繁琐；
  - 脚本化能力较弱，处理逻辑要靠用户空间工具或脚本。

### 2.2 核心概念

- `tracefs` 挂载点：`/sys/kernel/debug/tracing`
- 常用控制文件：
  - `current_tracer`
  - `available_tracers`
  - `set_ftrace_filter`
  - `trace` / `trace_pipe`
- 前端工具：
  - `trace-cmd`
  - `kernelshark`（图形界面）

### 2.3 典型使用场景

- 分析某个内核函数调用频率、延迟；
- 调试调度延迟、关中断时间过长；
- 粗粒度查看系统调用/内核路径。

## 3. perf：基于采样的性能分析工具

### 3.1 定位与特点

- `perf` 是 Linux 内核自带的 **性能计数器/采样分析工具**。
- 主要关注：
  - CPU 时钟、硬件 PMU 事件；
  - 指令 cache、LLC、分支预测等；
  - 支持内核态 + 用户态混合采样。
- 优点：
  - 适合 **性能剖析 (profiling)**，找 “哪段代码最耗 CPU”；
  - 支持 call graph（栈回溯）、火焰图等；
  - 通用性强，对应用和内核都能采样。
- 缺点：
  - 偏向统计采样，不适合需要精确事件序列的调试；
  - 对复杂场景，命令和参数较多，有学习成本。

### 3.2 常见命令模式

- `perf stat`：统计整体性能计数器（IPC、LLC-misses 等）
- `perf record + perf report`：采样并分析热点函数
- `perf top`：实时的热点函数视图

### 3.3 典型使用场景

- 应用/内核混合性能瓶颈定位（hotspot 分析）；
- 分析高 CPU / 短时间抖动问题；
- 和火焰图（FlameGraph）组合使用。

## 4. eBPF：可编程、高灵活度的新一代追踪框架

### 4.1 定位与特点

- eBPF 是在内核中运行的 **安全受控、可验证的小程序** 机制。
- 可以挂载到：
  - kprobe/kretprobe（内核函数）
  - tracepoint（静态探针）
  - uprobes（用户态函数）
  - 网络 XDP / tc 钩子等
- 优点：
  - **可编程**：支持在内核态执行一定逻辑（过滤、聚合、计数）；
  - 安全：有 verifier 校验，防止挂死内核；
  - 丰富的生态：BCC、bpftrace、各种现成工具。
- 缺点：
  - 学习成本较高：需要理解 BPF 程序模型、map、helper 等；
  - 内核版本依赖较强（老内核 eBPF 特性不完整）。

### 4.2 典型用户态框架

- BCC（BPF Compiler Collection）
  - Python/Lua 驱动 eBPF，脚本风格强；
  - 提供大量现成工具（`biolatency`, `execsnoop`, `tcptop` 等）。
- bpftrace
  - 类似 DTrace 的脚本语言；
  - 更适合交互式、实验型追踪。

### 4.3 典型使用场景

- 在线环境的细粒度性能观测：IO 延迟分布、TCP 重传、调度延迟；
- 只追踪某些 PID/进程、某些条件下的事件；
- 自定义可观测性平台（如云原生 observability）。

## 5. SystemTap：脚本化内核/用户态动态追踪

### 5.1 定位与特点

- SystemTap 提供一套 **脚本语言 + 运行时**，来动态插桩内核和用户态程序。
- 工作方式：
  - 根据脚本生成内核模块；
  - 加载模块后，按脚本逻辑收集事件、打印结果。
- 优点：
  - 功能非常强大，几乎可以在内核任意位置插桩；
  - 脚本语言表达力强，适合复杂逻辑。
- 缺点：
  - 安全性与稳定性风险较高（生成内核模块）；
  - 对内核调试符号依赖较强；
  - 部署门槛高，不太适合云环境/容器环境大规模使用。

### 5.2 典型使用场景

- 深入的内核问题调试（在受控测试环境）；
- 复杂事件关联分析。

## 6. sysdig：面向运维/安全的系统调用级追踪

### 6.1 定位与特点

- sysdig 是一个 **系统调用 + 事件审计** 工具，现在也有 eBPF 后端。
- 聚焦于：
  - 应用行为观测：文件访问、网络连接、进程生命周期；
  - 安全审计与故障排查。
- 优点：
  - 上手较友好，有类似 `tcpdump` 风格的过滤和输出；
  - 与容器/云原生结合紧密，Falco 等项目基于此生态。
- 缺点：
  - 粒度主要在 syscall/事件层，对于纯内核内部细节不如 ftrace/eBPF 灵活；
  - 更偏运维、安全而非深度性能调优。

### 6.2 典型使用场景

- 排查 “谁删了这个文件、谁占用了端口” 等审计类问题；
- 观察容器内应用的行为；
- 结合 Falco 做安全规则检测。

## 7. 对比总结：何时选谁？

| 框架       | 类型/侧重点                    | 适合场景                                   | 优点                          | 局限/注意点                                 |
| ---------- | ----------------------------- | ------------------------------------------ | ----------------------------- | ------------------------------------------- |
| ftrace     | 内核原生函数追踪              | 调度、函数调用、IRQ 延迟等                 | 内置、开销小、稳定            | 配置不够友好，脚本化弱                      |
| perf       | 采样型性能分析                | CPU hotspot、应用+内核性能剖析             | PMU 支持强、火焰图友好        | 偏统计，难得事件序列细节                    |
| eBPF (+BCC/bpftrace) | 可编程追踪/观测      | 生产环境细粒度观测、云原生可观测性         | 灵活、安全、生态工具丰富      | 学习曲线陡、依赖较新内核                    |
| SystemTap  | 脚本化动态插桩（编模块）      | 深度内核调试、复杂事件逻辑                 | 功能极强                      | 部署风险大，不适合生产、环境要求高          |
| sysdig     | 系统调用级事件 & 审计         | 运维排错、安全审计、容器行为观测           | 上手快、适配云原生            | 对内核内部细节能力有限，偏运维/安全         |

## 8. 选型建议（实践视角）

- **先 ftrace/perf，再 eBPF**：
  - 简单性能问题 → `perf stat/top/record`；
  - 调度/函数级问题 → ftrace / trace-cmd。
- **复杂线上观测 → eBPF (BCC / bpftrace)**：
  - 对 IO 延迟、网络行为、特定进程行为做按需追踪；
  - 用现成 eBPF 工具快速收集指标。
- **实验室深度内核调试 → SystemTap / 甚至直接改内核打印**。
- **偏运维/安全审计 → sysdig / Falco / eBPF 安全工具**。
