# Linux 内核网卡数据处理链路分析

source https://www.bilibili.com/video/BV1p4PjzCEGa?spm_id_from=333.788.videopod.sections
> 依据典型 Linux 网络栈流程整理，用于配合「Linux 内核网卡数据处理链路分析」视频学习记录。

## 1. 整体视角：从“网线”到用户态

自下而上的整体路径可以概括为：

- 网卡硬件（NIC）
- 驱动 & DMA / IOMMU（IOVA）
- 中断 / NAPI 轮询
- 内核协议栈（L2/L3/L4）
- 报文聚合/分发（GRO、RFS、RPS 等）
- 套接字接收队列 & 用户态 `recv()` / `read()`

发送方向则反过来：用户态 → 套接字发送队列 → 内核协议栈 → qdisc → 网卡驱动 → NIC DMA 发送。

后面章节按“接收路径”和“发送路径”分别展开。

---

## 2. 硬件与 DMA：从网卡到内存

### 2.1 网卡接收队列（RX Queue）

- 每个网卡通常有多个 **RX 队列**（per‑queue ring buffer）。
- 每个队列对应一组 **描述符 ring**，里面记录：
  - 用于 DMA 的地址（物理地址或 IOVA）
  - buffer 长度
  - 状态位（OWN、ERROR、CHECKSUM 等）。

### 2.2 DMA、IOMMU 与 IOVA

- **DMA**：网卡不通过 CPU 直接操作内存，通过总线发起 DMA 读写。
- **IOMMU** 存在时，驱动给 NIC 的地址通常是 **IOVA**（I/O Virtual Address）：
  - IOVA → IOMMU → 物理地址 PA。
  - 提供内存隔离、安全性及灵活内存布局。
- Linux 驱动通过 DMA API（如 `dma_map_single()`）把内存映射为 DMA 地址（可能就是 IOVA）。

## 3. 中断、NAPI 与 SoftIRQ

### 3.1 中断模式

- 传统：收一个包触发一次中断，性能差。
- 现代 NIC：**MSI-X + 中断合并**，一个队列对应一个中断向量，可绑定不同 CPU。

### 3.2 NAPI（New API）

- 典型流程：
  - 中断触发 → 驱动在中断处理函数里 **禁用该队列中断**，并调度 NAPI。
  - NAPI poll 回调在 SoftIRQ 上下文中跑：
    - 从 RX ring 里批量拉取报文。
    - 处理到一定数量或 ring 耗尽后返回。
  - 若未处理完，SoftIRQ 会继续调度；若处理完且没有新包，再重新打开中断。

- 优点：
  - 减少中断风暴。
  - 批量处理，提高缓存友好性。

### 3.3 SoftIRQ 与 `NET_RX_SOFTIRQ`

- 网络接收处理主要在 `NET_RX_SOFTIRQ` 中执行。
- 驱动通过调用 `napi_schedule()` 等将 NAPI 实例加入待处理队列。

## 4. 收包路径：从 NIC 到 Socket

### 4.1 驱动层处理

NAPI poll 中典型步骤：

1. 从 RX ring 取出描述符，得到 DMA 完成的 buffer。
2. 解除 DMA 映射，构造 `skb`：
   - 设置数据指针、长度、校验状态等。
3. 把 `skb` 交给内核通用入口：`netif_receive_skb()` 或 `napi_gro_receive()`。

### 4.2 GRO / LRO 等聚合

- **GRO（Generic Receive Offload）**：
  - 将多个属于同一流的报文在内核里聚合成一个大的 `skb`。
  - 减少后续协议栈处理次数。
- 部分硬件支持 **LRO**（Large Receive Offload），在网卡侧合并；GRO 是软件侧的通用版。

### 4.3 L2 / L3 / L4 处理

- **L2：`eth_type_trans()`** 识别协议类型（IPv4/IPv6/VLAN 等）。
- **L3：IP 层**：
  - 路由查找（本机 vs 转发）。
  - 分片/重组，TTL，IP 选项等。
- **L4：TCP / UDP / ICMP**：
  - 对应传输层协议处理。
  - TCP：状态机、拥塞控制、窗口更新等。

### 4.4 RPS / RFS 与多核分发

- **RPS（Receive Packet Steering）**：
  - 基于 hash（如 5 元组）将报文从某个 CPU 的 `NET_RX_SOFTIRQ` 分发到另一个 CPU 的 backlog，以改善负载均衡 / 缓存局部性。
- **RFS（Receive Flow Steering）**：
  - 结合 socket 归属的 CPU，使同一流的报文尽量被调度到处理该 socket 的 CPU 上。

### 4.5 套接字接收队列与用户态

- 找到对应的 `sock` 后，数据被挂入：
  - TCP：`sk_receive_queue` / `sk_backlog` 等。
  - UDP：`sk_receive_queue`。
- 应用通过 `recv() / recvmsg()` 等系统调用从 socket buffer 取数据：
  - 系统调用从用户态进入内核。
  - 拷贝或 `zero-copy`（如 `sendfile`、`io_uring` 某些模式）数据到用户缓冲区。
  - 返回用户态。

## 5. 发包路径：从应用到网卡

### 5.1 用户态到内核

- 应用调用 `send()/sendto()/sendmsg()`：
  - 数据从用户缓冲区拷贝到内核 `skb`。
  - 填写目的地址、端口等元数据。

### 5.2 协议栈处理与路由

- L4：
  - TCP：组装 segment、维护序列号、窗口等。
  - UDP：构造 UDP 报文头。
- L3：
  - 路由查找，选出 egress 设备和下一跳。
  - 填写 IP 头部，计算校验。

### 5.3 qdisc 与流控

- 报文进入 **qdisc（队列规则）**：
  - `pfifo_fast`（经典）。
  - `fq_codel`、`fq` 等现代算法。
- 在 qdisc 阶段进行排队、整形、丢弃策略等。

### 5.4 驱动与 NIC 发送队列（TX Queue）

- qdisc 出队后，将 `skb` 交给驱动：
  - 驱动将 buffer 进行 DMA 映射（得到 DMA 地址/IOVA）。
  - 填写 TX ring 描述符。
  - 通知网卡硬件（写 doorbell 寄存器等）。
- 网卡通过 DMA 从内存中读取报文，并发出去。

---

## 6. 性能与调优要点概览

- **多队列 + RSS**：根据 5 元组在多个 RX/TX 队列之间分散负载。
- **中断亲和性（IRQ affinity）**：绑定队列到固定 CPU，配合 RFS 让同一流在同一核上处理。
- **NAPI 权重**：控制每次 poll 处理的报文数，在延迟和吞吐之间平衡。
- **GRO / GSO**：减少协议栈调用次数。
- **IOMMU / IOVA 策略**：在安全与性能之间做权衡（如是否使用直通模式）。

---

## 7. 小结

- 接收方向：**NIC → DMA(IOVA) → 中断/NAPI → SoftIRQ → 协议栈 → socket → 用户态**。
- 发送方向：**用户态 → socket → 协议栈 → qdisc → 驱动/DMA → NIC**。
- 核心思想：
  - 通过 **多队列、多核、批处理（NAPI/GRO）、地址虚拟化（IOVA）** 达到高性能。
  - 通过 **IOMMU + IOVA** 提供设备 DMA 的安全隔离和灵活内存管理能力。
