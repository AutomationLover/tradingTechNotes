## 1. Interrupt Basics

### 1.1 `/proc/interrupts`

- Contains statistics related to interrupts on the system
- **Column 1:** IRQ number (only shows interrupts with installed handlers)
- **Column 2:** Counter of interrupts received (per CPU)
- **Column 3/4:** Type of interrupt and device (e.g., XT-PIC, IO-APIC)
- **Column 5:** Device name (supplied by `devname` parameter to `request_irq()`)

**Edge vs. Level Triggered:**
- **IO-APIC-edge:** Edge-triggered interrupts
- **IO-APIC-fasteoi:** Level-triggered; EOI = End Of Interrupt (acknowledged in PIC)

### 1.2 `/proc/irq` and SMP IRQ Affinity

- **SMP IRQ affinity:** Ability to assign IRQs to specific CPUs
- **`/proc/irq/IRQ_NUMBER/smp_affinity`:** Hex bitmask for CPU cores
- **`/proc/irq/IRQ_NUMBER/smp_affinity_list`:** CPU list format
- **`/proc/irq/default_smp_affinity`:** Default affinity for non-active IRQs
- Use `echo 1 > /proc/irq/1/smp_affinity` to pin an IRQ to CPU 0

## 2. Interrupt Handler Registration

### 2.1 `request_irq()` and `free_irq()`

**Header:** `<linux/interrupt.h>`

```c
int request_irq(unsigned int irq,
                irq_handler_t handler,
                unsigned long flags,
                const char *name,
                void *dev);
```

**Parameters:**
- **irq:** Interrupt number (hard-coded for legacy devices, probed for others)
- **handler:** Function pointer: `typedef irqreturn_t (*irq_handler_t)(int, void *);`
- **flags:** Bitmask of interrupt options
- **name:** Display name in `/proc/interrupts`
- **dev:** Used for shared interrupt lines; must match when calling `free_irq()`

**Return:** 0 on success, non-zero on failure (e.g., `-EBUSY` if IRQ already in use)

```c
void free_irq(unsigned int irq_no, void *dev);
```

- Must pass the **same** `dev` pointer used in `request_irq()` when releasing

### 2.2 Interrupt Flags

| Flag | Description |
|------|-------------|
| **IRQF_SHARED** | Interrupt can be shared with other devices; required if IRQ already has a handler |
| **IRQF_ONESHOT** | Interrupt not re-enabled after handler finishes; **mandatory** for threaded interrupts; keeps line disabled until threaded handler runs |
| **IRQF_PROBE_SHARED** | Probe for shared interrupt capability |

### 2.3 Return Value of Interrupt Handlers

```c
typedef irqreturn_t (*irq_handler_t)(int, void *);
```

- **IRQ_NONE:** Interrupt was not from this device or was not handled
- **IRQ_HANDLED:** Interrupt was handled by this device

### 2.4 Device Identification in Shared IRQs

- Devices with interrupt support have a **status register** that can be read in the handler
- Example: 8250 serial port uses **IIR (Interrupt Information Register)** to verify the interrupt source

## 3. Interrupt Context

### 3.1 Process vs. Interrupt Context

- **Process context:** Kernel executing on behalf of a process (e.g., system call)
- **Interrupt context:** Kernel executing an interrupt handler; **not backed by a process**

### 3.2 Restrictions in Interrupt Context

- **Cannot sleep** — no `kmalloc(GFP_KERNEL)`, `ssleep()`, semaphores, etc.
- **`current`** points to the interrupted process
- Use `in_interrupt()` to check if code is running in interrupt context

---

## 4. Enabling and Disabling Interrupts

### 4.1 Why Disable Interrupts?

- Guarantees an interrupt handler will not preempt current code
- Disabling interrupts also disables kernel preemption
- **Note:** Does not protect against concurrent access from another CPU — use locks for that

### 4.2 APIs

| Function | Description |
|----------|-------------|
| **`local_irq_disable()`** | Unconditionally disables interrupts — **dangerous** if interrupts were already disabled |
| **`local_irq_enable()`** | Unconditionally enables interrupts |
| **`local_irq_save(flags)`** | Saves interrupt state to `flags` and disables interrupts |
| **`local_irq_restore(flags)`** | Restores previous interrupt state |

**Best practice:** Use `local_irq_save()` / `local_irq_restore()` to preserve prior state.

## 5. Top Half and Bottom Half

### 5.1 Goals of Interrupt Handlers

1. **Execution time:** Handler must run quickly; long handlers can slow the system and lose interrupts
2. **Execution context:** Handlers run in hard-interrupt context with CPU-local interrupts disabled — locking and sleeping must be avoided

### 5.2 Two-Part Design

| Part | Responsibility |
|------|----------------|
| **Top Half (Hard IRQ)** | Acknowledge interrupt, copy data from device, schedule bottom half |
| **Bottom Half (Soft IRQ)** | Remaining deferred work |

**Example (Network Card):**
- **Top half:** Acknowledge interrupt, copy packets to memory, push to protocol layer, ready card for more packets, schedule bottom half
- **Bottom half:** Rest of packet processing

### 5.3 Bottom Half Mechanisms

1. **Softirqs**
2. **Tasklets**
3. **Workqueues**

## 6. Threaded Interrupt Handlers

### 6.1 Concept

- Alternative to formal bottom-half mechanisms
- Minimizes time with interrupts disabled by moving most work into a kernel thread
- Kernel automatically schedules the threaded handler — no manual bottom-half scheduling

### 6.2 `request_threaded_irq()`

```c
int request_threaded_irq(unsigned int irq,
                         irq_handler_t handler,
                         irq_handler_t thread_fn,
                         unsigned long irqflags,
                         const char *devname,
                         void *dev_id);
```

- **handler:** Quick hardware acknowledgment; wakes thread
- **thread_fn:** Runs in process context; can sleep
- Thread priority: `MAX_USER_RT_PRIO/2` (higher than regular processes)
- **IRQF_ONESHOT** is mandatory for threaded IRQs (keeps line disabled until thread runs)

## 7. GPIO to IRQ Mapping (Raspberry Pi)

- Input GPIOs can be used as IRQ signals (edge or level triggered)
- **`gpio_to_irq(unsigned gpio)`** maps GPIO number to IRQ number
- Use the returned IRQ with `request_irq()`

**Raspberry Pi push-button setup:** Connect button to GPIO input pin, other side to 3.3V via 330Ω resistor.

## 8. Softirqs

### 8.1 Overview

- Bottom halves that run at high priority **with hardware interrupts enabled**
- **Implementation:** `kernel/softirq.c`
- **Header:** `<linux/softirq.h>`

### 8.2 Softirq Types (from `include/linux/interrupt.h`)

| Index | Name | Purpose |
|-------|------|---------|
| HI_SOFTIRQ | High-priority tasklets | |
| TIMER_SOFTIRQ | Timers | |
| NET_TX_SOFTIRQ | Network transmit | |
| NET_RX_SOFTIRQ | Network receive | |
| BLOCK_SOFTIRQ | Block layer | |
| IRQ_POLL_SOFTIRQ | IRQ polling | |
| TASKLET_SOFTIRQ | Regular tasklets | |
| SCHED_SOFTIRQ | Scheduler | |
| HRTIMER_SOFTIRQ | High-res timers | |
| RCU_SOFTIRQ | RCU processing | |

- Number of softirqs is **statically determined** at compile time

### 8.3 Registration and Execution

- **`open_softirq(nr, action)`** — Register handler at runtime
- **`raise_softirq(nr)`** — Mark softirq pending (uses `local_irq_save`, `raise_softirq_irqoff`, `local_irq_restore`)
- **Per-CPU bitmap:** `irq_stat[smp_processor_id()].__softirq_pending`
- **`do_softirq()`** / **`__do_softirq()`** — Execute pending softirqs

### 8.4 When Are Softirqs Run?

1. After hard interrupt handlers complete (with IRQ lines enabled) — `irq_exit()` from `do_IRQ()`
2. On `local_bh_enable()` or `spin_unlock_bh()`
3. When `ksoftirqd/n` kernel thread is awakened

### 8.5 ksoftirqd

- After many iterations over the softirq bitmap, kernel defers to **ksoftirqd/n** (one per CPU)
- Runs in process context so it can be preempted by user processes
- Avoids processor monopolization by softirqs

### 8.6 Key Properties

- **Preemption:** A softirq never preempts another softirq; only interrupt handlers can preempt
- **Parallelism:** Same softirq can run on different CPUs simultaneously
- **Sleep:** Softirq handlers **cannot sleep**
- **Priority:** Kernel iterates bitmap LSB first; lower index = higher priority

### 8.7 Disabling Softirqs

- **`local_bh_disable()`** — Disable softirq and tasklet processing on local CPU
- **`local_bh_enable()`** — Enable (calls can be nested; only final enable actually enables)

## 9. Tasklets

### 9.1 Overview

- Bottom-half mechanism built on top of softirqs (HI_SOFTIRQ and TASKLET_SOFTIRQ)
- **Easier to use** than raw softirqs; **dynamically allocatable**
- **Header:** `<linux/interrupt.h>`

### 9.2 Data Structure

```c
struct tasklet_struct {
    struct tasklet_struct *next;
    unsigned long state;      // TASKLET_STATE_SCHED, TASKLET_STATE_RUN
    atomic_t count;           // 0 = enabled, nonzero = disabled
    void (*func)(unsigned long);
    unsigned long data;
};
```

### 9.3 State and Count

- **TASKLET_STATE_SCHED:** Tasklet is scheduled to run
- **TASKLET_STATE_RUN:** Tasklet is running (multiprocessor protection)
- **count = 0:** Tasklet enabled; **count ≠ 0:** Tasklet disabled

### 9.4 Scheduling

- **`tasklet_schedule(t)`** — Schedule regular tasklet
- **`tasklet_hi_schedule(t)`** — Schedule high-priority tasklet
- Per-CPU lists: `tasklet_vec` (TASKLET_SOFTIRQ), `tasklet_hi_vec` (HI_SOFTIRQ)
- Same tasklet scheduled twice before running → runs **only once**
- **Interrupts are enabled** when tasklet runs (softirqs run with interrupts enabled)

### 9.5 `tasklet_kill()`

```c
void tasklet_kill(struct tasklet_struct *t);
```

- Removes tasklet from pending queue
- **Must not be called from interrupt context** — it can sleep
- Waits for tasklet to finish if already running

### 9.6 Tasklet vs. Softirq

| Aspect | Softirqs | Tasklets |
|--------|----------|----------|
| Allocation | Compile time | Dynamic |
| Reentrancy | Same softirq can run on different CPUs | Same tasklet cannot run on multiple CPUs |

## 10. Workqueues

### 10.1 Overview

- **Only choice when bottom half needs to sleep** (I/O, mutexes, semaphores)
- Work items executed by **worker threads** in **process context**
- **Implementation:** `kernel/workqueue.c`
- **Header:** `<linux/workqueue.h>`

### 10.2 Data Structures

- **Work queue:** `struct workqueue_struct`
- **Work item:** `struct work_struct` with `work_func_t func`

### 10.3 Worker Threads (kworker)

- **Naming:** `kworker/%u:%d%s` (CPU, id, priority)
- **CPU-bound:** `kworker/<core>:<id>`
- **Unbound:** `kworker/u<pool>:<id>` — can run on any CPU
- Use `taskset -p [PID]` to check CPU affinity
- Use `cat /proc/$(pidof kworker)/stack` to inspect stack

### 10.4 System vs. Dedicated Workqueues

- **System workqueue:** `system_wq` — shared; timing not predictable
- **Dedicated workqueue:** `alloc_workqueue()` — own queue and control

### 10.5 Key APIs

| Function | Purpose |
|----------|---------|
| **`schedule_work(work)`** | Queue work on system workqueue |
| **`queue_work(wq, work)`** | Queue work on specific workqueue |
| **`queue_work_on(cpu, wq, work)`** | Queue on specific CPU |
| **`cancel_work_sync(work)`** | Cancel work (waits if running) |
| **`flush_work(work)`** | Wait for work to finish |
| **`flush_workqueue(wq)`** | Wait for all work on queue |
| **`schedule_delayed_work(dwork, delay)`** | Schedule with delay (jiffies) |
| **`cancel_delayed_work(dwork)`** | Cancel delayed work |
| **`flush_delayed_work(dwork)`** | Cancel timer and run work immediately |

### 10.6 Workqueue Flags

| Flag | Description |
|------|-------------|
| **WQ_UNBOUND** | Work can run on any CPU |
| **WQ_HIGHPRI** | High-priority worker pool |
| **WQ_SYSFS** | Visible in sysfs |
| **WQ_FREEZABLE** | Participates in system suspend |
| **WQ_MEM_RECLAIM** | May be used in memory reclaim; ensures rescuer thread |
| **WQ_CPU_INTENSIVE** | CPU-intensive; won't block other work |

### 10.7 `alloc_ordered_workqueue()`

- Ordered workqueue: at most one work item at a time, in queued order
- Implemented as unbound workqueue with `max_active = 1`

## 11. Comparison: Softirqs vs. Tasklets vs. Workqueues

| Aspect | Softirqs | Tasklets | Workqueues |
|--------|----------|----------|------------|
| **Execution context** | Interrupt | Interrupt | Process |
| **Reentrancy** | Yes (different CPUs) | No (same tasklet) | Yes |
| **Sleep** | Cannot | Cannot | Can |
| **Preemption** | No | No | Yes |
| **Ease of use** | Hard | Easy | Easy |
| **When to use** | Scalability/speed, no sleep | No sleep needed | Need to sleep |

- [ ] Register and free interrupt handlers with `request_irq()` / `free_irq()`
- [ ] Use `in_interrupt()` to detect interrupt context
- [ ] Use `local_irq_save()` / `local_irq_restore()` for safe interrupt disabling
- [ ] Implement threaded interrupt handlers with `request_threaded_irq()`

- [ ] Configure SMP IRQ affinity when needed
